Two small but meaningful releases back to back. **1.0.18** simplifies the server configuration system, and **1.0.19** fixes a critical SDK bug affecting servers running multiple licensed plugins simultaneously.

## 1.0.18 - Server Config Simplification

### 🗂️ The Old Structure

Previously, the server generated a nested folder structure on first run:

```
diogenes-config/
├── server/
│   └── config.yml
└── discord/
    └── messages.yml
```

This made sense in theory, but in practice it was unnecessary overhead. The `messages.yml` file contained Discord embed messages that are better handled at the Sinope layer, and the extra folders added friction for anyone setting up the server quickly.

---

### ✅ The New Structure

Starting from `1.0.18`, the server generates a single file at the root, right next to the JAR:

```
DiogenesServer.jar
config.yml          ← generated here on first run
```

The config file itself was also cleaned up. Two unused fields (`admin-role-id` and `log-channel-id`) were removed:

```yaml
server:
  port: 8080
  mongo-uri: "mongodb://localhost:27017"

discord:
  bot-token: "YOUR_BOT_TOKEN_HERE"
```

---

### 🖥️ First Boot Messages

The first-run console output now follows the same visual standard as the rest of the server logs:

**Before:**
```
[!] Config folder created on: /home/user/diogenes-config
[!] Please fill the information in ./diogenes-config/server/config.yml and restart.
```

**After:**
```
[ INFO ] CONFIG: File created at: /home/user/config.yml
[ INFO ] CONFIG: Please fill config.yml and restart the server.
```

---

### ⚠️ Migration Note

If you are upgrading from a previous version, your old `diogenes-config/server/config.yml` will **not** be read automatically. Before restarting with `1.0.18`:

1. Copy your `mongo-uri` and `bot-token` values from the old file.
2. Delete the `diogenes-config/` folder.
3. Run the new JAR — it will generate a fresh `config.yml` at the root.
4. Paste your credentials and restart.

## 1.0.19 — SDK Singleton Fix

### 🐛 The Problem

When two or more plugins using the Diogenes SDK were loaded on the same server, they shared the same SDK instance — because `DiogenesSDK` was implemented as a Kotlin `object` (singleton).

This caused the second plugin to overwrite the `pluginFolder` of the first, making both plugins read from the same `license.yml` file. In practice, this meant the second plugin could authenticate using the first plugin's key without having its own license configured.

```
[ INFO ] AUTH: Successfully authenticated. PRODUCT: plugin1     ← correct
[ INFO ] AUTH: Successfully authenticated. PRODUCT: plugin2     ← wrong, used plugin1's key
```

---

### ✅ The Fix

`DiogenesSDK` was refactored from an `object` to a `class`. Each `init()` call now creates an isolated instance with its own `productId`, `baseUrl`, and `pluginFolder`.

```
plugins/Plugin1/license.yml     ← read by plugin1's instance
plugins/Plugin2/license.yml     ← read by plugin2's instance
```

The public API is **unchanged** — your `onEnable()` code stays exactly the same:

```java
DiogenesSDK.init(this, "your-product-id", "http://your-server:8080", () -> { ... });
```

---

### 🎨 Banner Improvement

The ASCII banner now prints together with the authentication result, instead of appearing separately at the top of the log:

```
__________________________________________________________________________________

    _____^_
   ...
   / \_/   0  \        Diogenes 1.0.19

[ INFO ] AUTH: Successfully authenticated. PRODUCT: plugin
__________________________________________________________________________________
```

If multiple plugins are loaded, only the first one prints the banner. Subsequent plugins print only their result line — keeping the console clean.
